## 一、PF7-1：一种水网运行操作系统架构及其资源管理方法

### 1.1 八维度评分

| 维度 | 评分 | 评审意见 |
|------|------|----------|
| 新颖性 | 7.0 | 将OS分层抽象思想引入水网领域有一定新颖性，但四层架构（感知-模型-决策-执行）本质上是经典CPS/SCADA架构变体，"操作系统"命名更多是概念包装而非本质创新。缺乏与华为鸿蒙/工业互联网OS等真实OS架构的技术对比。 |
| 创造性 | 6.5 | "优先级抢占+时间片配额+带宽保底"混合调度策略是操作系统领域成熟方案的直接移植，未说明水网场景的特殊适配难点。WNAL降级函数 $\Lambda$ 形式化过于简单（仅 $L_i \to L_{i-1}$），缺少降级决策的判别逻辑。调度目标函数 $\min J$ 是标准多目标加权和形式，无特殊创造性。 |
| 充分公开 | 7.0 | 四层架构有概念描述但缺少层间接口的具体数据结构与协议定义。统一API仅给出元数据六元组 $\Psi$ 的字段名，未给出API调用的请求/响应格式、序列化方式。资源调度的求解算法（如何从目标函数得到调度决策）未公开。 |
| 权利要求完整性 | 6.5 | 独立权利要求1过于功能化——"构建四层架构""建立统一API""执行实时调度""执行WNAL降级"均为功能描述，缺少实现这些功能的具体技术手段/数学特征。一个本领域技术人员仅凭权利要求1无法区分本发明与已知CPS平台架构。权利要求12（系统权利要求）仅列出模块名，无结构关系。 |
| 实施例充分性 | 7.5 | 三个实施例覆盖部署、性能、降级三个方面，场景选择合理。实施例1给出具体工程规模（61渠池、72闸站等），实施例2有定量对比数据（丢包率0.83%→0.12%），实施例3有降级时间指标。但参数来源不明，未说明是仿真还是实测。 |
| 技术方案清晰度 | 7.0 | S1-S6六步骤逻辑清晰，从架构到调度到降级到闭环，层次分明。但步骤间的数据流和控制流描述不足——例如S3中资源向量 $\mathbf{R}(t)$ 如何驱动S4的调度决策，缺少算法流程。健康度 $H$ 如何反馈到S4的参数更新也未明确。 |
| 保护范围适当性 | 5.5 | 独立权利要求保护范围过宽——几乎涵盖任何分层+统一接口+资源调度+降级的水利软件系统。缺少区分性技术特征。同时权利要求11"适用于长距离输水、多源多汇调配及城市水网"是应用场景限定而非技术特征，在中国专利法下保护力弱。从属权利要求2-10虽有细化但仍偏功能化。 |
| 工程可实施性 | 7.5 | 实施例参数合理（12.6万条/秒吞吐、38-64ms联锁时延），与大型输水工程实际规模匹配。降级恢复时间2.4s→5min可信。但未说明HydroOS的技术栈选型（操作系统内核、消息中间件、数据库等），工程实施指导性不足。 |

**总分: 54.5/80**

### 1.2 问题清单

**P0（必须修改）**：

1. **独立权利要求严重功能化**：权利要求1的四个特征均为功能描述，缺少具体技术手段。建议至少将资源调度的混合策略（优先级抢占+时间片配额+带宽保底）和降级触发条件写入独立权利要求。
2. **缺少现有技术对比表**：背景技术仅做定性描述，未提供与具体现有技术（如OpenPLC、FIWARE、Azure IoT、已公开的水利SCADA专利）的结构化对比表，审查员无法判断区别特征。
3. **资源调度目标函数缺少求解方法**：$\min J$ 仅给出目标函数和约束，未说明采用何种优化算法（启发式/整数规划/规则引擎）求解，不满足充分公开要求。
4. **缺少核心算法伪代码**：整个技术方案无一处伪代码，资源调度器、降级状态机、健康度评估闭环的实现逻辑不透明。

**P1（建议修改）**：

1. 背景技术第4段提到"数据中台或数字孪生平台"，建议引用具体已知方案（如三峡智慧水利平台、水利部数字孪生先行先试项目），增加可检索性。
2. 服务元数据 $\Psi$ 的六个字段（id, type, latency, priority, qos, security）定义过于简略，建议补充字段取值范围和数据类型。
3. 健康度 $H$ 中的三个分量 $A_{avail}$, $R_{recover}$, $S_{stability}$ 缺少定义公式。
4. 实施例数据应标注来源（仿真/实测/估算），增强可信度。
5. 权利要求11的场景限定建议删除或移入说明书。

## 二、PF7-2：一种水网运行数据实时采集、质量评估与融合处理方法

### 2.1 八维度评分

| 维度 | 评分 | 评审意见 |
|------|------|----------|
| 新颖性 | 6.5 | 多源数据融合、数据质量评估、卡尔曼滤波融合均为成熟技术。四维质量评估（完整性、一致性、时效性、精度）在数据治理领域已有ISO 25012等标准定义。核心创新点——质量评分耦合观测噪声协方差 $R_k = R_0 \cdot \exp(\gamma(1-Q_{all}))$ 有一定新颖性，但指数映射并非唯一或最优选择，缺乏该映射函数的理论依据。 |
| 创造性 | 6.5 | 将质量评分与卡尔曼滤波噪声参数动态耦合属于工程组合创新，但自适应卡尔曼滤波（如Sage-Husa滤波）本身就具备在线估计噪声协方差的能力，本方案的额外贡献需要更清晰论证。分层缺失修复（短→插值、长→预测）是常见工程实践。 |
| 充分公开 | 7.0 | 卡尔曼融合公式和质量自适应映射有明确数学表达。但四维质量各分量 $Q_c, Q_k, Q_t, Q_a$ 的计算方法未给出——仅有符号无计算式，技术人员无法复现质量评估环节。权重 $w_i$ 的确定方法（先验/自适应）也未说明。 |
| 权利要求完整性 | 6.5 | 独立权利要求1列出六个步骤但均为功能描述："建立统一接入""执行时间同步""构建四维质量模型""执行异常检测""执行自适应融合""闭环更新"——缺少任何数学/逻辑特征。权利要求7仅说"采用卡尔曼滤波或其扩展形式"，保护范围极宽但实质内容空泛。权利要求8（噪声协方差随质量评分动态调节）是最具区分性的特征，但被放在了从属权利要求中。 |
| 实施例充分性 | 7.5 | 三个实施例覆盖接入评估、修复融合、分级运行三个场景。数据丰富（48.6万条异常、97.2%修复率、水位误差0.064→0.038m）。实施例3的复合故障场景设计合理。但缺少与现有方法（如标准卡尔曼、无质量自适应融合）的严格对照实验。 |
| 技术方案清晰度 | 7.5 | S1-S6流程清晰，从接入→对齐→评估→修复→融合→输出层层递进。质量自适应融合的核心公式表述明确。但S4中"规则阈值+统计检验+时序模型"联合检测缺少具体算法选择和判决逻辑。 |
| 保护范围适当性 | 6.0 | 独立权利要求保护的是"多源接入+质量评估+异常修复+融合+闭环"的通用流程，范围过宽。核心区分特征（质量评分→噪声协方差映射）仅在从属权利要求8中体现。建议将权利要求8的内容上提到独立权利要求中。 |
| 工程可实施性 | 7.5 | 数据规模（1.5万+SCADA测点）和性能指标（120ms延迟、99.98%可用率）与实际工程匹配。协议适配器支持OPC-UA、MQTT等主流工业协议，工程可行。但质量评估模型的参数标定（$w_i$, $\gamma$, 阈值）方法未给出工程指导。 |

**总分: 55.0/80**

### 2.2 问题清单

**P0（必须修改）**：

1. **四维质量分量缺计算公式**：$Q_c, Q_k, Q_t, Q_a$ 只有名称无计算方法，这是全文技术方案的核心环节之一，必须给出每个分量的数学定义（如 $Q_c = 1 - N_{miss}/N_{total}$）。
2. **独立权利要求功能化严重**：建议将"观测噪声协方差根据综合质量评分通过指数映射动态调节"（现权利要求8内容）写入独立权利要求。
3. **缺少现有技术对比表**：背景技术未引用任何具体的已知数据融合专利或标准（如ISO 25012数据质量框架、已公开的SCADA数据清洗专利），审查员无法确认新颖性。
4. **缺少核心算法伪代码**：质量自适应融合的完整计算流程（从原始数据到融合输出的逐步算法）应以伪代码形式呈现。

**P1（建议修改）**：

1. $R_k = R_0 \cdot \exp(\gamma(1-Q_{all}))$ 中 $\gamma$ 的物理意义和取值范围应明确说明。
2. 实施例2的"缺失修复成功率97.2%"需定义"成功"的判据。
3. 时间同步 $\tau_i' = \tau_i + \Delta\tau_i$ 过于简略，应补充偏移量 $\Delta\tau_i$ 的估计方法（NTP/PTP/GPS等）。
4. 权重 $w_1, w_2, w_3, w_4$ 的确定方法（AHP/专家打分/数据驱动）应补充说明。
5. 建议在实施例中增加与"不使用质量自适应"的标准卡尔曼滤波的定量对照数据。

## 三、PF7-3：一种面向水网的可视化运行监控与人机交互方法

### 3.1 八维度评分

| 维度 | 评分 | 评审意见 |
|------|------|----------|
| 新颖性 | 6.0 | 水网拓扑可视化、多尺度缩放、自然语言交互均为已知技术方向。GIS+SCADA叠加渲染在电力、燃气等行业已有成熟应用。物理校核判定调度建议可行性有一定新意，但"建议→仿真→验证"模式在工业决策支持系统中并不罕见。整体属于已知技术在水网领域的集成应用。 |

| 权利要求完整性 | 6.0 | 独立权利要求1的六个特征均为功能描述，技术实质空洞。"构建拓扑语义图并映射实时数据"——任何监控系统都要做这件事；"执行一张图动态渲染"——这是目标而非手段；"对自然语言指令进行意图解析并任务编排"——这是ChatGPT级别的通用描述。缺乏任何区分性技术特征。 |
| 实施例充分性 | 7.0 | 三个实施例场景设置合理（渲染联动、交互校核、告警处置）。给出了定量指标（异常定位时间132s→49s、建议可执行率96.4%、$Q_{hmi}$ 从0.82→0.90）。但缺少系统架构的技术实现细节（用什么前端框架、什么图形引擎、什么NLP模型）。 |
| 技术方案清晰度 | 6.5 | S1-S6逻辑流程清晰，从拓扑构建→渲染→缩放→交互→校核→优化。但多个步骤停留在功能描述层面，技术方案的"how"严重不足。 |
| 保护范围适当性 | 5.0 | 独立权利要求范围极宽——任何具备可视化+交互+模型验证的水利监控系统都可能落入保护范围，但由于缺少具体技术特征，实际保护力极弱（容易被无效）。建议收窄到水网领域的具体技术方案。 |
| 工程可实施性 | 7.0 | 实施例参数（59渠池、2.1万测点、1s刷新、1800条术语词典、320ms解析时延）合理可信。但作为一件方法专利，缺少软件架构、技术选型和部署方案的描述，工程指导性不足。 |

**总分: 49.0/80**

### 3.2 问题清单

**P0（必须修改）**：

1. **独立权利要求极度功能化，几乎无技术实质**：六个特征均为"做什么"而非"怎么做"。这是本批次最严重的权利要求问题。建议至少将多尺度语义保持的数据结构/算法、物理校核的约束模型写入独立权利要求。
2. **缺少现有技术对比表**：未引用任何已知可视化平台专利或产品（如ABB Ability, Siemens MindSphere的SCADA可视化、国内智慧水务平台专利），新颖性无法判定。
3. **核心算法/数据结构完全缺失**：语义保持映射 $\Phi$、渲染函数 $\mathcal{R}$、意图解析 $\mathcal{P}$ 均只有符号无内容，不满足充分公开要求。必须补充至少一个核心环节的完整算法。
4. **缺少核心算法伪代码**：全文无任何伪代码。
5. **缺少结构化对照实验数据**：实施例2提到"可执行率96.4%较传统经验式建议提升约22%"，但对照组的定义、实验设置、统计方法均未说明。

**P1（建议修改）**：

1. 交互质量指标 $Q_{hmi}$ 的三个分量（$Acc$, $T_{resp}^{-1}$, $Sat$）中，$Sat$（满意度）为主观指标，建议补充客观化度量方法或说明评价流程。
2. 物理校核应具体说明调用哪类模型（水动力模型/简化模型/规则引擎）及校核时间约束。
3. 权利要求8（交互方式包括文本/语音/模板）是显而易见的常规选择，保护价值低，建议替换为更具技术含量的特征。
4. 建议在背景技术中增加对电力/燃气等行业已有可视化监控系统的横向对比。

## 六、PF7批次共性缺陷总结

### 共性缺陷1：独立权利要求严重功能化（四件均存在，P0级）

四件专利的独立权利要求1均呈现相同模式：罗列3-6个功能步骤，每个步骤用"构建/建立/执行/输出"等动词+功能描述，完全没有写入**实现该功能的具体技术手段**（如特定算法、数据结构、数学模型、判决逻辑）。

**后果**：
- 审查员容易以"技术方案不清楚"或"缺少必要技术特征"驳回；
- 即便授权，保护范围因过于宽泛（谁都能设计不同的实现方式落入范围）而极易被无效；
- 独立权利要求无法体现发明的核心技术贡献。

**建议**：每件专利的独立权利要求中至少应包含1-2个具有区分性的**数学/逻辑/结构特征**。例如：
- PF7-1：将"优先级抢占+时间片配额+带宽保底"混合策略和WNAL降级状态机写入；
- PF7-2：将质量评分→噪声协方差指数映射 $R_k = R_0 \cdot \exp(\gamma(1-Q_{all}))$ 写入；
- PF7-3：将多尺度语义保持映射的具体数据结构和物理约束校核模型写入；
- PF7-4：将三元组数据结构定义和安全约束A/B验证机制写入。

### 共性缺陷2：缺少现有技术对比表（四件均存在，P0级）

四件专利的背景技术部分均采用"概述性批评现有技术不足"的写法，未引用任何具体的已知专利、产品或标准。每件文末虽附有"现有技术检索方向"但明确标注"未实际检索"。

**后果**：
- 新颖性和创造性论证缺乏支撑；
- 审查员无法判定区别特征是否真正具有新颖性；
- 背景技术部分不符合"客观描述现有技术"的撰写规范。

**建议**：每件专利应补充至少3项最接近现有技术的具体引用（专利号/论文/产品名），并以表格形式列出与本发明的技术差异对比。

### 共性缺陷3：缺少核心算法伪代码（四件均存在，P0级）

四件专利均给出了一定程度的数学公式（目标函数、状态方程等），但**没有一件**提供核心算法的伪代码。公式给出了"优化什么"，但没说明"怎么算"。

**后果**：
- 充分公开要求不满足——技术人员看到公式但无法按步骤实施；
- 技术方案的可重复性无法保证。

**建议**：每件专利至少提供1段核心伪代码（10-30行），覆盖最关键的计算/决策流程：
- PF7-1：资源调度器主循环；
- PF7-2：质量自适应融合主循环；
- PF7-3：自然语言→结构化意图→物理校核→输出的完整流水线；
- PF7-4：三元组→聚类→规则→策略更新的自进化主循环。

### 共性缺陷4：缺少结构化对照实验数据（四件均存在，P1级）

四件专利的实施例均给出了性能数据（时延降低、误差减小、效率提升等），但：
- 缺少明确的**对照组定义**（"传统方法"具体指什么系统/配置？）；
- 缺少实验的**控制变量说明**；
- 缺少数据的**统计显著性分析**（标准差、置信区间等）；
- 未标注数据来源是仿真/实测/估算。

**建议**：每件专利至少在1个实施例中增加结构化的对照实验表（包含基线方法名称、相同测试条件、关键指标对比、数据来源标注）。

### 共性缺陷5：自评审分数虚高（四件均存在，信息性问题）

四件专利的内置自评审分数为9.1-9.2/10，与本次外部评审结果（49.0-55.0/80，折算6.1-6.9/10）差距显著。自评审流程在"形式完整性"方面有效，但在"技术实质性"（算法深度、权利要求质量、现有技术对比）方面的审查力度不足。

### 共性缺陷6：PF7四件专利间的差异化不足

PF7四件专利均围绕HydroOS平台展开，技术方案框架高度相似（六步法、闭环优化、WNAL分级）。特别是：
- PF7-1的"健康度闭环优化"与PF7-4的"自进化闭环"存在概念重叠；
- PF7-2的"数据质量评估"与PF7-3的"交互质量评估"方法论类似；
- 四件专利均包含"闭环反馈+周期更新"机制，但各自的闭环对象和更新逻辑没有清晰区分。

**建议**：在每件专利的背景技术中增加一段说明本发明与同族其他专利的关系和差异。
