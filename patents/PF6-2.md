# PF6-2：一种水网控制系统硬件在环测试方法及系统

## （一）技术领域

本发明属于智慧水利控制测试与工业自动化验证技术领域，具体涉及一种面向CHS (Cybernetics of Hydro Systems) 与HydroOS平台的水网控制系统硬件在环测试（HIL, Hardware-in-the-Loop）方法及系统，涵盖真实PLC/RTU/SCADA接入、仿真-硬件实时交互、时间同步与故障注入验证。

## （二）背景技术

模型在环测试（MIL）能够在纯软件环境验证控制逻辑，但无法充分暴露真实硬件链路中的时延、抖动、丢包、I/O扫描周期、协议栈开销等工程问题。水网控制系统上线后常见问题并非算法本身错误，而是"硬件执行节拍与仿真节拍不一致""协议交互拥塞""边缘设备响应受限"等系统级耦合故障。

现有水网调试通常采用现场联调方式，存在风险高、窗口短、扰动不可控等缺陷。缺乏标准化HIL测试平台时，工程团队难以在上线前量化硬件与控制策略匹配性，导致现场试错成本高。特别在分层分布式控制与安全联锁场景下，毫秒级触发路径若未经HIL验证，可能在突发事件中出现动作迟滞。

当前技术主要问题包括：第一，**硬件与仿真接口不统一**，不同协议与设备驱动耦合复杂；第二，**时间同步机制不足**，仿真时间与硬件时钟漂移导致闭环误判；第三，**故障注入能力弱**，难以系统测试延迟、丢包、设备失效等异常；第四，**评估标准不一致**，难以形成可审计准入依据。

因此，需要一种面向水网控制系统的HIL测试方法及系统，使真实硬件在可控环境下与仿真对象闭环运行，系统化评估实时性、可靠性与安全性。

## （三）发明内容

### 1. 技术问题

本发明要解决的技术问题是：针对现有控制验证无法覆盖真实硬件行为、缺少统一时间同步与故障注入机制的问题，提出一种水网控制系统硬件在环测试方法及系统，实现硬件接入条件下的闭环可验证测试。

### 2. 技术方案

本发明提供一种水网控制系统硬件在环测试方法及系统，包括如下步骤：

**步骤S1：构建HIL系统架构。**

系统包括：
1) 实时仿真器模块 $P_r$；
2) 被测硬件模块 $H$（PLC/RTU/边缘控制器/SCADA站）；
3) 协议适配模块 $A$（OPC-UA/Modbus/TCP/IEC 104）；
4) 时间同步模块 $T_s$；
5) 故障注入模块 $F_i$；
6) 评估与归档模块 $E$。

**步骤S2：建立硬件-仿真实时交互回路。**

仿真器输出过程量 $y_k$ 作为硬件输入，硬件输出控制量 $u_k$ 回写仿真器，形成闭环：
$$
u_k=H(y_k),\quad x_{k+1}=P_r(x_k,u_k,d_k)
$$

**步骤S3：执行时间同步与步长协调。**

采用主从时钟同步机制，定义仿真步长 $\Delta t_s$ 与硬件扫描周期 $\Delta t_h$ 协调条件：
$$
|\Delta t_s-\Delta t_h|\le \epsilon_t
$$
当偏差超阈值时触发步长重整或缓冲补偿。

时间同步与步长补偿算法伪代码：

```
算法：HIL时间同步与步长补偿
输入：仿真步长dt_s, 硬件扫描周期dt_h, 同步阈值eps_t, 缓冲区大小B
输出：同步后的闭环执行序列

1. 初始化时钟偏差累积器 drift = 0
2. 初始化环形缓冲区 buffer[B]，写指针wp=0，读指针rp=0
3. 同步主时钟 t_master = 0

4. WHILE 测试未结束 DO
5.     // 仿真器侧：按dt_s推进
6.     t_sim = t_master
7.     y_k = P_r.output(x_k)
8.     buffer[wp] = (t_sim, y_k)           // 写入缓冲
9.     wp = (wp + 1) mod B
10.
11.    // 硬件侧：按dt_h读取
12.    t_hw = hardware.read_clock()
13.    drift = t_sim - t_hw
14.
15.    IF |drift| <= eps_t THEN
16.        // 正常同步：直接传递
17.        u_k = H.read_output()
18.    ELSE IF |drift| <= 3 * eps_t THEN
19.        // 缓冲补偿：从缓冲区插值
20.        y_interp = linear_interpolate(buffer, t_hw)
21.        H.write_input(y_interp)
22.        WAIT_FOR hardware response
23.        u_k = H.read_output()
24.        drift_compensated = drift_compensated + 1
25.    ELSE
26.        // 步长重整：调整仿真步长对齐
27.        dt_s_new = dt_s + sign(drift) * min(|drift|/2, dt_s*0.1)
28.        P_r.set_timestep(dt_s_new)
29.        LOG_WARNING("步长重整: dt_s调整为" + dt_s_new)
30.        u_k = H.read_output()  // 使用最近有效值
31.    END IF
32.
33.    // 仿真推进
34.    x_{k+1} = P_r.step(x_k, u_k, d_k)
35.    t_master = t_master + dt_s
36.    k = k + 1
37. END WHILE
```

**步骤S4：进行故障注入测试。**

在闭环运行中注入可配置故障：
- 通信故障：延迟 $\tau$、丢包率 $p_{loss}$、乱序；
- 设备故障：传感器漂移、执行器卡滞、模块掉线；
- 计算故障：CPU过载、任务超时。

**步骤S5：建立性能评估指标体系。**

定义HIL关键指标：
$$
J=[L_{rt},P_{loss},T_{recover},R_{safe},E_{ctrl},S_{stab}]
$$
分别表示实时时延、丢包率、故障恢复时间、安全违规率、控制代价和稳定性评分。

稳定性评分计算：
$$
S_{stab} = 1 - \frac{\sum_{k}\mathbb{1}[|y_k - r_k| > \delta_{stab}]}{N_{total}}
$$

**步骤S6：通过判定与测试报告生成。**

当 $R_{safe}=0$ 且指标满足阈值时判定通过，否则输出未通过原因、薄弱环节与整改建议。报告含场景配置、故障轨迹、关键曲线和版本信息，纳入审计档案。

### 3. 有益效果

1. 在真实硬件参与下验证控制系统闭环行为，提升上线可信度；
2. 通过时间同步机制降低仿真-硬件节拍失配风险；
3. 通过故障注入系统化评估鲁棒性与恢复能力；
4. 通过统一指标与判定规则形成标准化验收依据；
5. 支持HydroOS控制链路的工程级准入测试。

## （五）具体实施方式

### 实施例1：PLC控制器HIL接入验证

选取某输水段控制站，接入两台PLC与一台RTU作为被测硬件，实时仿真器采用明渠水动力模型，仿真步长 $\Delta t_s=100\,ms$，硬件扫描周期 $\Delta t_h=100\,ms$。

通过OPC-UA与Modbus/TCP双协议接入，执行24 h闭环测试。结果：
- 实时时延95分位 $L_{rt}=82\,ms$；
- 安全违规率 $R_{safe}=0$；
- 控制性能与MIL基线偏差 < 4.5%。

判定：满足上线前HIL要求。

### 实施例2：通信时延与丢包故障注入

注入通信延迟 $\tau=300\sim900\,ms$ 与丢包率 $p_{loss}=5\%\sim20\%$。系统监测到异常后触发降级策略，关键控制通道抢占带宽，非关键数据降频。

结果显示：
- 在 $p_{loss}=15\%$ 条件下仍维持安全约束零违规；
- 故障恢复时间 $T_{recover}$ 中位数 38 s；
- 相比无降级机制基线，恢复时间缩短约34%。

### 实施例3：设备故障与时间漂移联合场景

模拟水位传感器偏置 + 执行器响应迟滞 + 时钟漂移。时间同步模块检测 $|\Delta t_s-\Delta t_h|>\epsilon_t$ 后执行步长补偿，故障注入模块记录全过程。

在该场景中系统完成自动重同步，稳定性评分 $S_{stab}$ 维持在0.86以上，未出现联锁误动作。

### 对照实验：本发明HIL方法与传统验证方式对比

在同一输水控制系统上，分别采用三种验证方式进行对照实验：

| 对比指标 | 现场联调(传统方式) | 纯MIL仿真验证 | 本发明HIL闭环平台 |
|---------|------|------|--------|
| 硬件参与 | 真实硬件+真实工程 | 无硬件 | 真实硬件+仿真工程 |
| 测试风险 | 高(影响实际输水) | 无 | 低(仿真环境) |
| 时间同步验证 | 不可控 | 不涉及 | 主从同步+偏差补偿 |
| 通信故障覆盖 | 仅自然发生的故障 | 不涉及 | 系统化注入6类故障 |
| 发现的硬件相关问题 | 3个(事后发现) | 0个 | 8个(事前发现) |
| 恢复时间(通信抖动场景) | 58 s(无降级) | N/A | 38 s(含降级机制) |
| 测试可重复性 | 不可重复 | 完全可重复 | 完全可重复 |
| 单次测试周期 | 3-5天(含协调) | 2小时 | 4小时(含硬件接入) |
| 安全违规事件 | 1次(现场超限) | 0 | 0 |

实验表明：本发明方法在保持真实硬件参与的前提下，将硬件相关问题发现率提升约167%，故障恢复时间缩短约34%，同时消除了现场联调的安全风险。

## （六）权利要求书

**权利要求1.** 一种水网控制系统硬件在环测试方法及系统，其特征在于，包括以下步骤：

步骤一，构建由实时仿真器模块 $P_r$、被测硬件模块 $H$、协议适配模块 $A$、时间同步模块 $T_s$、故障注入模块 $F_i$ 和评估归档模块 $E$ 组成的HIL测试平台；

步骤二，通过协议适配模块将仿真器输出过程量 $y_k$ 写入被测硬件，将硬件输出控制量 $u_k$ 回写仿真器，建立 $u_k=H(y_k)$，$x_{k+1}=P_r(x_k,u_k,d_k)$ 的实时闭环交互；

步骤三，采用主从时钟同步机制实时监测仿真步长 $\Delta t_s$ 与硬件扫描周期 $\Delta t_h$ 的偏差，当偏差 $|\Delta t_s-\Delta t_h|$ 不超过阈值 $\epsilon_t$ 时直接传递数据，当偏差超过 $\epsilon_t$ 但不超过 $3\epsilon_t$ 时从环形缓冲区插值补偿，当偏差超过 $3\epsilon_t$ 时执行仿真步长重整；

步骤四，在闭环运行中按故障配置表注入通信故障、设备故障和计算故障；

步骤五，基于性能指标 $J=[L_{rt},P_{loss},T_{recover},R_{safe},E_{ctrl},S_{stab}]$ 评估闭环行为；

步骤六，当 $R_{safe}=0$ 且其余指标满足阈值时判定通过，否则输出薄弱环节与整改建议。

**权利要求2.** 根据权利要求1所述的方法，其特征在于，所述被测硬件包括PLC、RTU、边缘控制器或SCADA站。

## （七）摘要

本发明公开了一种水网控制系统硬件在环测试方法及系统。该方法在实时仿真器与真实PLC/RTU/SCADA硬件之间建立闭环交互，通过协议适配实现数据实时交换，并采用主从时钟同步机制配合三级偏差处理策略（直接传递、缓冲插值补偿、步长重整）协调仿真步长与硬件扫描周期；在闭环过程中注入通信延迟、丢包、传感器漂移、执行器卡滞等故障，基于实时时延、恢复时间、安全违规率和稳定性评分等指标进行统一评估；当安全违规率为零且指标满足阈值时判定通过，否则输出薄弱环节与整改建议。该方法可显著提升水网控制系统上线前工程验证的真实性、可重复性与可追溯性。
